# 1. 정의 (Definition)
- 프로젝트가 **이해관계자가 기대하는 가치와 성과물을 실제로 인도**하는 것과 관련된 활동. 
- "무엇을 만들어야 하는가"와 "그게 제대로 만들어졌는가"에 집중하는 영역.

# 2. 요구사항 관리 (Requirements Management)
## 2-1. **요구사항 유형 6가지**

|유형|설명|
|---|---|
|**Business Requirements**|조직이 왜 이 프로젝트를 하는가 (비즈니스 목적)|
|**Stakeholder Requirements**|이해관계자 관점의 필요사항|
|**Solution Requirements - Functional**|시스템/제품이 무엇을 해야 하는가|
|**Solution Requirements - Non-functional**|성능, 보안, 신뢰성 등 품질 기준|
|**Transition Requirements**|현재→미래 상태로 전환 시 필요한 것|
|**Project Requirements**|프로젝트 자체의 제약/조건|

## 2-2. **요구사항 추적 매트릭스 (Requirements Traceability Matrix, RTM)**

- 요구사항이 어디서 왔고(출처), 어디에 반영됐는지(인도물) 추적
- 범위 변경 시 영향 분석에 활용

# 3. 범위 관리 (Scope)

|구분|내용|
|---|---|
|**Product Scope**|제품/서비스의 기능과 특성|
|**Project Scope**|제품을 만들기 위해 수행해야 할 작업|

> 💡 Product Scope = **무엇을** 만드나 / Project Scope = **어떻게** 만드나

## 3-1. **인수 기준 (Acceptance Criteria)**

- 인도물이 완료됐다고 인정받기 위한 조건
- 이해관계자와 사전 합의 필수

# 4. 인도 방식별 완료 기준
|방식|완료 기준|
|---|---|
|**예측형 (Waterfall)**|Acceptance Criteria 충족 → 공식 인수|
|**애자일 (Agile)**|Definition of Done (DoD) 충족|

### 4-1. **Definition of Done (DoD)**

- 스프린트/이터레이션 단위로 "완료"를 팀이 정의한 기준
- 예: 코드 리뷰 완료 + 테스트 통과 + 문서화 완료


# 5. 가치 인도 방식

|개념|설명|
|---|---|
|**MVP** (Minimum Viable Product)|핵심 기능만 담은 최소 제품. 빠른 피드백 수집 목적|
|**MBI** (Minimum Business Increment)|비즈니스 가치를 전달하는 최소 단위의 증분|
|**점진적 인도 (Incremental Delivery)**|완성된 부분부터 단계적으로 인도|
|**반복적 개발 (Iterative Development)**|피드백 반영하며 점점 개선|


# 6. 범위 확인 vs 품질 통제

- 시험에서 혼동 주의!

|구분|수행 주체|목적|
|---|---|---|
|**Validate Scope** (범위 확인)|이해관계자|인도물 공식 인수/승인|
|**Control Quality** (품질 통제)|프로젝트 팀|인도물이 기준 충족하는지 검사|

> 순서: **Control Quality 먼저 → 그 다음 Validate Scope**


# 7. 성과 확인 (Outcomes)

- ✅ 이해관계자 요구사항이 충족된 인도물 생성
- ✅ 프로젝트 목표가 실현됨
- ✅ 프로젝트 기간 동안 가치가 지속적으로 인도됨
- ✅ 인수 기준/DoD가 명확하게 정의되고 달성됨


# 8. **시험 포인트 요약**

- Product Scope vs Project Scope 구분
- RTM: 요구사항 출처~인도물까지 추적
- Validate Scope(이해관계자 인수) vs Control Quality(팀 검사) 순서
- MVP = 최소 기능 제품, MBI = 최소 비즈니스 가치 단위
- 예측형 → Acceptance Criteria / 애자일 → DoD
