# 1. 한 줄 요약
- 프로젝트가 계획 대비 어디에 있는지 파악하고, 예측하고, 개선하는 도메인. 
- Predictive는 EVM으로, Agile은 Velocity/Burndown으로 측정.


# 2. 핵심 활동 (Key Activities)
- 측정 지표(KPI) 설정
- 성과 데이터 수집 및 분석
- 성과 보고 (대시보드, 상태 보고서)
- 예측 (Forecasting) — 완료 시 예산/일정 예측
- 측정 결과 기반 의사결정 및 시정 조치

# 3. 주요 개념 & 도구

## 3-1. EVM (Earned Value Management) — Predictive 핵심

- **기본 3개 값 먼저 이해:**

|용어|약자|의미|한 줄 요약|
|---|---|---|---|
|**Planned Value**|PV|지금까지 **계획된** 작업의 예산|"이 시점에 얼마 쓸 계획이었나"|
|**Earned Value**|EV|지금까지 **완료된** 작업의 예산|"실제로 얼마치 일을 했나"|
|**Actual Cost**|AC|지금까지 **실제로 사용한** 비용|"실제로 얼마를 썼나"|
|**BAC**|BAC|전체 프로젝트 **총 승인 예산**|"처음에 얼마로 계획했나"|

### 3-1-1. 편차 (Variance) — 현재 상태

| 공식                                   | 의미    | 양수(+)    | 음수(-)   |
| ------------------------------------ | ----- | -------- | ------- |
| **SV (Schedule Variance) = EV - PV** | 일정 편차 | 일정 앞서감 ✅ | 일정 지연 ❌ |
| **CV (Cost Variance) = EV - AC**     | 원가 편차 | 예산 절감 ✅  | 예산 초과 ❌ |


### 3-1-2. 성과 지수 (Performance Index) — 효율성

| 공식                | 의미    | 1보다 크면   | 1보다 작으면 |
| ----------------- | ----- | -------- | ------- |
| **SPI = EV / PV** | 일정 효율 | 일정 앞서감 ✅ | 일정 지연 ❌ |
| **CPI = EV / AC** | 원가 효율 | 예산 절감 ✅  | 예산 초과 ❌ |

> **암기 팁:** EV가 분자. SV/CV는 빼기, SPI/CPI는 나누기.


### 3-1-3. 예측 (Forecasting) — 앞으로 어떻게 될지

|공식|의미|
|---|---|
|**EAC = BAC / CPI**|현재 효율 그대로 가면 최종 얼마 들까 (가장 많이 쓰임)|
|**ETC = EAC - AC**|지금부터 완료까지 얼마 더 필요한가|
|**VAC = BAC - EAC**|최종적으로 예산 초과/절감 예상 금액|
|**TCPI = (BAC - EV) / (BAC - AC)**|남은 작업을 원래 예산 안에 끝내려면 얼마나 효율적이어야 하나|

> **TCPI 해석:** 1보다 크면 지금보다 더 효율적으로 해야 함 (빡셈), 1보다 작으면 여유 있음.

---

**예제 문제:**

> BAC = 100만원, 현재 시점에서 PV = 50만원, EV = 40만원, AC = 45만원일 때:
> 
> - CV = 40 - 45 = **-5만원** (예산 초과)
> - SV = 40 - 50 = **-10만원** (일정 지연)
> - CPI = 40 / 45 = **0.89** (예산 초과 중)
> - SPI = 40 / 50 = **0.8** (일정 20% 뒤처짐)
> - EAC = 100 / 0.89 = **약 112만원** (최종 112만원 들 것으로 예측)

## 3-2. Agile 측정 지표

### 3-2-1. Velocity (속도)
- 팀이 Sprint마다 완료한 **Story Point** 수
- 몇 번 Sprint 돌려보면 평균 Velocity 나옴 → 다음 Sprint 계획에 활용
- **시험 포인트:** "Velocity가 갑자기 떨어졌다" → 장애물 존재 가능성. Scrum Master가 원인 파악해야 함.

### 3-2-2. Burndown Chart (번다운 차트)

- X축: 시간 (Sprint 기간), Y축: 남은 작업량
- 이상적으로는 **우상단 → 우하단으로 내려와야** 함
- **시험 포인트:** "번다운 차트가 평평하다" → 진척 없음. 장애물 있다는 신호.

**Burnup Chart (번업 차트)**

- 번다운과 반대 — 완료된 작업량을 올라가는 방향으로 표시
- 범위 변경(Scope Change)을 시각화하기 좋음

**Lead Time vs Cycle Time (Kanban)**

| |**Lead Time**|**Cycle Time**|
|---|---|---|
|**측정 구간**|요청 접수 → 완료|작업 시작 → 완료|
|**포함 범위**|대기 시간 포함|실제 작업 시간만|

## 3-3. 측정 함정 (Measurement Pitfalls)

측정이 잘못되면 오히려 해가 됨. 시험에 나오는 함정들:

- **허영 지표 (Vanity Metrics):** 실제 가치와 무관한 숫자만 좋아 보임. 예: 코드 줄 수, 회의 수.
- **목표 왜곡:** 측정 지표 달성이 목적이 되면 실제 가치 훼손. 예: "이슈 수 줄이기" 위해 이슈 등록 안 함.
- **과도한 측정:** 측정 자체에 너무 많은 시간 소모.

# 4. 다른 도메인과의 연결
- **3-1 개발 접근방식:** Predictive → EVM, Agile → Velocity/Burndown. 접근방식에 따라 측정 도구 달라짐.
- **3-5 계획:** PV(계획 가치)는 계획 도메인에서 나옴. 계획이 부실하면 EVM도 의미 없음.
- **3-8 불확실성:** EAC/TCPI가 안 좋으면 리스크 대응 계획 재검토.

# 5. 시험 포인트

- **"CPI = 0.8이다"** → 예산 초과 중. 1달러 쓸 때마다 0.8달러치 일만 하고 있음.
- **"SPI = 1.2다"** → 일정 앞서가는 중. 계획보다 20% 더 진척됨.
- **"EAC 구하라"** → **BAC / CPI** 공식 쓰면 됨 (별도 조건 없을 때).
- **"프로젝트 종료 시 예산 초과 예상이다, 어떻게 하나"** → **스폰서에게 보고 + 원인 분석 + 시정 조치 계획.** 혼자 해결하려 하면 오답.
- **"Agile 팀에서 EVM 쓰나"** → 쓸 수 있지만 일반적으로 Velocity/Burndown이 더 적합.
- **"번다운 차트가 Sprint 초반에 평평하다"** → 초반 계획/분석에 시간 쓰는 거라 정상일 수 있음. 중후반에도 평평하면 문제.
