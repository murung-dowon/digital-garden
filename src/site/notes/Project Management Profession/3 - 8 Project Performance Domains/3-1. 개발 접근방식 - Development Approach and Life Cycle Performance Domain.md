>[!tip] "프로젝트 성격에 맞게 개발 방식을 골라라. Agile이 무조건 좋은 것도, Waterfall이 나쁜 것도 아니다."

## 1. 한 줄 요약
- 프로젝트 특성에 맞는 개발 방식(Predictive/Agile/Hybrid)을 선택하고, 그에 맞는 생애주기를 설계하는 것. 시험의 50%가 여기서 나옴.

## 2. 핵심 활동 (Key Activities)
- 프로젝트 특성 분석해서 개발 접근방식 선택
- 생애주기(Phase) 정의 및 단계별 목표 설정
- 인도 주기(Delivery Cadence) 결정
- 접근방식에 맞는 프로세스 테일러링

## 3. 주요 개념 & 도구
### 3-1. 개발 접근방식 3가지

| |**Predictive (예측형)**|**Agile (적응형)**|**Hybrid (혼합형)**|
|---|---|---|---|
|**별칭**|Waterfall, 폭포수|Adaptive, 애자일|-|
|**요구사항**|초반에 명확히 정의|변경 가능, 점진적 구체화|일부 확정 + 일부 유동|
|**결과물 인도**|프로젝트 끝에 한 번|짧은 주기로 반복 인도|혼합|
|**리스크**|후반에 발견 시 대응 어려움|짧은 주기로 빠른 발견/수정|-|
|**고객 참여**|초반/후반 집중|전 기간 지속적 참여|-|
|**적합한 상황**|요구사항 명확, 규제 강함, 대형 인프라|요구사항 불명확, 변경 잦음, 혁신적 제품|대부분의 실제 프로젝트|

**접근방식 선택 기준 (시험에 자주 나옴):**

| 상황                        | 추천 접근방식         |
| ------------------------- | --------------- |
| 요구사항이 명확하고 변경 가능성 낮음      | Predictive      |
| 요구사항이 불명확하고 고객 피드백 필요     | Agile           |
| 일부는 확정, 일부는 탐색 필요         | Hybrid          |
| 규제/컴플라이언스가 강한 산업 (제약, 건설) | Predictive      |
| 소프트웨어, 신제품 개발             | Agile 또는 Hybrid |

### 3-2. 인도 주기 (Delivery Cadence)
- 결과물을 언제, 얼마나 자주 인도하느냐:

|유형|설명|예시|
|---|---|---|
|**Single Delivery**|프로젝트 끝에 한 번만|다리 건설|
|**Multiple Deliveries**|여러 번 나눠서 인도|기능별 출시|
|**Periodic Deliveries**|정해진 주기로 인도|2주 스프린트|
|**Continuous Delivery**|수시로 배포|CI/CD 파이프라인|
- **CI (Continuous Integration):** 개발자들이 코드를 매일 여러 번 공유 저장소에 올리고, 올릴 때마다 자동으로 빌드/테스트가 돌아가는 것. "합치면서 생기는 충돌을 빠르게 발견"하는 게 목적.
- **CD (Continuous Delivery/Deployment):** 테스트 통과한 코드를 자동으로 배포하는 것.

### 3-3. Scrum 프레임워크 (Agile의 대표 주자)

**3가지 역할:**

|역할|책임|
|---|---|
|**Product Owner (PO)**|백로그 관리, 우선순위 결정, 비즈니스 가치 극대화|
|**Scrum Master (SM)**|팀이 Scrum 잘 쓰도록 지원, 장애물 제거. 서번트 리더십|
|**Development Team**|실제 개발. 자기조직화(Self-organizing), 크로스펑셔널|

**Scrum 이벤트 5가지:**

|이벤트|목적|주기|
|---|---|---|
|**Sprint**|실제 개발 기간|1~4주|
|**Sprint Planning**|이번 Sprint에 뭘 할지 계획|Sprint 시작|
|**Daily Scrum**|어제 뭐 했고, 오늘 뭐 할지, 장애물 뭔지|매일 15분|
|**Sprint Review**|완성된 결과물 이해관계자에게 시연|Sprint 끝|
|**Sprint Retrospective**|팀 프로세스 개선점 논의|Sprint 끝 (Review 후)|

**Scrum 산출물 3가지:**

|산출물|설명|
|---|---|
|**Product Backlog**|전체 해야 할 것들 목록. PO가 관리|
|**Sprint Backlog**|이번 Sprint에 할 것들. 팀이 선택|
|**Increment**|Sprint 완료 후 나온 동작하는 결과물|


### 3-4. Kanban (흐름 기반 Agile)

Scrum처럼 Sprint 없이 **흐름(Flow)** 기반으로 관리:

- **WIP 제한 (Work In Progress Limit):** 동시에 진행 중인 작업 수를 제한해서 병목 방지
- **칸반 보드:** To Do → In Progress → Done 시각화
- **리드 타임:** 작업 요청부터 완료까지 걸리는 시간

**Scrum vs Kanban:**

| |Scrum|Kanban|
|---|---|---|
|주기|Sprint (고정)|흐름 (유동)|
|역할|PO/SM/Dev Team 명확|역할 유연|
|변경|Sprint 중 변경 제한|언제든 추가 가능|
|적합|신규 개발|유지보수, 운영|


## 4. 다른 도메인과의 연결

- **3-2 이해관계자:** Agile은 이해관계자가 Sprint Review마다 참여. Predictive는 주요 단계별 참여.
- **3-3 팀:** Agile은 Self-organizing 팀. Scrum Master가 서번트 리더십 발휘.
- **3-4 측정:** Agile은 Velocity, Burndown Chart로 측정. Predictive는 EVM.
- **3-5 계획:** Predictive는 초반에 상세 계획. Agile은 Rolling Wave Planning (점진적 계획).

## 5. 시험 포인트

- "요구사항이 자주 바뀐다" → **Agile 또는 Hybrid** 선택이 정답.
- "규제가 엄격한 산업이다" → **Predictive** 선택.
- "백로그 우선순위를 PM이 바꿨다" → **오답.** PO가 하는 일.
- "Scrum Master가 팀원에게 일을 할당했다" → **오답.** SM은 지시하지 않음. 팀이 자율적으로 선택.
- "Sprint 중간에 고객이 새 기능을 요청했다" → **Product Backlog에 추가하고 다음 Sprint에서 우선순위 결정.** 현재 Sprint 중단 아님.
- "Daily Scrum에서 기술적 문제를 깊게 토론하기 시작했다" → **오답.** Daily Scrum은 15분, 상태 공유만. 깊은 논의는 별도 미팅으로.
- "Retrospective랑 Review 헷갈리면?" → **Review = 결과물 시연 (이해관계자 참여), Retrospective = 팀 내부 프로세스 개선 (팀만).**
