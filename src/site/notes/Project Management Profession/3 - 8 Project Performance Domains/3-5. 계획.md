> [!tip] "계획은 한 번 만들고 끝이 아니라, 프로젝트 내내 살아있는 문서다."

# 1. 한 줄 요약

- 범위/일정/원가/품질/리스크 등 모든 영역의 계획을 수립하고 통합하는 도메인. 
- 계획 자체가 목적이 아니라 가치 인도를 위한 수단임.

# 2. 핵심 활동

- 프로젝트 관리 계획서 (PMP) 수립 및 통합
- 범위 정의 및 WBS 작성
- 일정 개발 (네트워크 다이어그램, 주공정 파악)
- 원가 산정 및 예산 수립
- 리스크 계획 → **2-10 참조**
- 품질 계획 → **2-8 참조**
- 변경 통제 프로세스 수립

# 3. 주요 개념 및 도구 

## 3-1. 프로젝트 관리 계획서 (Project Management Plan)

모든 하위 계획들을 통합한 최상위 문서. "이 프로젝트 어떻게 관리할 거냐"의 모든 답이 여기 있어야 함.

포함되는 하위 계획들 (Subsidiary Plans):

| 한국어 | English |
|---|---|
| 범위 관리 계획 | Scope Management Plan |
| 일정 관리 계획 | Schedule Management Plan |
| 원가 관리 계획 | Cost Management Plan |
| 품질 관리 계획 | Quality Management Plan |
| 자원 관리 계획 | Resource Management Plan |
| 의사소통 관리 계획 | Communications Management Plan |
| 리스크 관리 계획 | Risk Management Plan |
| 조달 관리 계획 | Procurement Management Plan |
| 이해관계자 참여 계획 | Stakeholder Engagement Plan |
| 변경 관리 계획 | Change Management Plan |
| 형상 관리 계획 | Configuration Management Plan |

**시험 포인트:** "PM이 범위 변경을 혼자 승인했다" → 오답. 변경 관리 계획에 따라 **CCB 승인** 받아야 함.


## 3-2. 범위 관리

### 3-2-0. Define Scope → Project Scope Statement

범위 관리의 첫 단계. "이 프로젝트에서 무엇을 만들 것인가"를 공식 문서로 정의함.

**output: Project Scope Statement** 구성 요소:

| 구성 요소 | 설명 | 예시 |
|---|---|---|
| **Product Scope Description** | 만들 제품/서비스 설명 | "iOS/Android 모바일 앱" |
| **Deliverables** | 인도물 목록 | 앱 바이너리, 설계 문서, 테스트 결과 |
| **Acceptance Criteria** | 완료 인정 기준 | "버그 0건, 응답속도 2초 이내" |
| **Project Exclusions** | 범위 제외 항목 | "웹 버전은 이번 프로젝트 범위 아님" |
| **Constraints** | 제약사항 | 예산 $50,000, 6개월 이내 완료 |
| **Assumptions** | 가정사항 | "고객이 서버 인프라 제공한다고 가정" |

> **시험 포인트:** Scope Statement ≠ Milestone List. 마일스톤은 Define Activities의 output. Scope Statement는 "뭘 만들지"이고, Milestone은 "언제 체크포인트 찍을지"라 레벨이 다름.

### 3-2-1. WBS (Work Breakdown Structure)
- 전체 작업을 관리 가능한 작은 단위로 쪼갠 계층적 구조
- 최하위 단위 = **Work Package** (실제 작업 할당 가능한 수준)
- WBS는 **산출물(Deliverable) 중심**으로 분해. 활동(Activity) 중심 아님.

### 3-2-2. 범위 기준선 (Scope Baseline)
- Scope Statement + WBS + WBS Dictionary 묶음
- 변경 통제 없이 바꾸면 안 됨

### 3-2-3. Gold Plating vs Scope Creep:

| |Gold Plating|Scope Creep|
|---|---|---|
|**발생 원인**|PM/팀이 요청 없이 기능 추가|승인 없이 범위가 슬금슬금 커짐|
|**문제**|자원 낭비, 가치 불명확|일정/예산 초과|
|**PMP 입장**|둘 다 ❌ 하면 안 됨|둘 다 ❌ 하면 안 됨|


## 3-3. 일정 관리

### 3-3-0. 일정 개발 프로세스 순서

일정은 아래 순서로 만들어짐. 각 단계의 output이 다음 단계의 input이 됨:

```
Define Scope          → Project Scope Statement ("뭘 만들지" 정의)
       ↓
WBS 작성              → Work Package 단위로 분해
       ↓
Define Activities     → Activity List + Activity Attributes + Milestone List ⭐
       ↓
Sequence Activities   → 네트워크 다이어그램 (활동 순서/의존관계)
       ↓
Estimate Durations    → Duration Estimates (각 활동 기간 산정)
       ↓
Develop Schedule      → CPM 적용 → 최종 Project Schedule
```

#### 📌 네트워크 다이어그램 작성 방법 (Diagramming Methods)

| 방법                                                  | 다른 이름                   | 핵심                      | 소프트웨어 채택 |
| --------------------------------------------------- | ----------------------- | ----------------------- | -------- |
| **PDM (Precedence Diagramming Method, 선후관계 도형 기법)** | AON (Activity-On-Node)  | 박스(노드) = 활동, 화살표 = 의존관계 | ✅ 대부분 사용 |
| **ADM (Arrow Diagramming Method, 화살표 도형 기법)**       | AOA (Activity-On-Arrow) | 화살표 = 활동, 원 = 이벤트       | ❌ 거의 안 씀 |

> 📝 **시험 포인트:** "대부분의 PM 소프트웨어가 쓰는 방법" → **PDM (AON)**. MS Project, Primavera 등 전부 PDM 기반.

##### PDM의 4가지 의존관계 (Dependency Types)

| 유형 | 설명 | 빈도 |
|------|------|------|
| **FS (Finish-to-Start)** | A 끝나야 B 시작 | 가장 흔함 ⭐ |
| **SS (Start-to-Start)** | A 시작해야 B 시작 가능 | 중간 |
| **FF (Finish-to-Finish)** | A 끝나야 B도 끝낼 수 있음 | 중간 |
| **SF (Start-to-Finish)** | A 시작해야 B가 끝남 | 거의 안 나옴 |

> ⚠️ **AOA의 한계:** FS 관계만 표현 가능. 복잡한 논리 표현 시 **Dummy Activity(점선 화살표)** 추가 필요 → 복잡해짐. 그래서 PDM이 표준이 됨.

> ❌ **자주 나오는 함정:**
> - "Critical Diagramming" → 존재하지 않는 용어
> - "Leads and Lags" → 기법이 아니라 PDM 안에서 쓰는 시간 조정 개념 (Lead = 선행, Lag = 지연)

#### 📌 Milestone (마일스톤)
- **기간이 0인** 특정 완료 시점. 작업이 아니라 체크포인트.
- 예: "설계 완료", "1차 납품", "테스트 시작"
- **Milestone List** = Define Activities의 공식 output
- Define Scope의 output(Project Scope Statement)에 마일스톤 언급은 있을 수 있지만, **공식 Milestone List 문서는 Define Activities에서 나옴**

> 📝 **시험 포인트 (Set A):** "마일스톤 목록은 어느 단계의 output?" → **Define Activities**. Define Scope 아님.

### 3-3-1. 주공정법 (Critical Path Method, CPM)
네트워크 다이어그램에서 가장 긴 경로 = **Critical Path (주공정)**

- Critical Path = 프로젝트 최소 완료 기간
- Critical Path 위의 활동은 **Float(여유시간) = 0**
- Float = 0 → 이 활동 늦어지면 **프로젝트 전체 지연**

#### 📦 Activity Node 구조

각 활동은 아래처럼 6칸짜리 박스로 표현됨:

```
┌─────────────────────┬─────────────────────┐
│  ES (최조착수일)     │  EF (최조완료일)     │
├─────────────────────┴─────────────────────┤
│              Activity Name                 │
│              Duration: X일                 │
├─────────────────────┬─────────────────────┤
│  LS (최지착수일)     │  LF (최지완료일)     │
└─────────────────────┴─────────────────────┘
```

| 약어 | 풀네임 | 의미 |
|------|--------|------|
| **ES** | Early Start | 가장 **빨리** 시작 가능한 날 |
| **EF** | Early Finish | 가장 **빨리** 끝낼 수 있는 날 |
| **LS** | Late Start | 지연 없이 **가장 늦게** 시작해도 되는 날 |
| **LF** | Late Finish | 지연 없이 **가장 늦게** 끝내도 되는 날 |

#### 🔁 Forward Pass (순방향 계산) → Early 날짜 계산

**시작 → 끝 방향**으로 흘러가며 ES/EF 계산

```
EF = ES + Duration - 1
다음 활동의 ES = 이전 활동의 EF + 1
(여러 선행 활동이 있으면 → EF 중 가장 큰 값 + 1)
```

#### 🔁 Backward Pass (역방향 계산) → Late 날짜 계산

**끝 → 시작 방향**으로 거슬러 올라가며 LS/LF 계산

```
LS = LF - Duration + 1
이전 활동의 LF = 다음 활동의 LS - 1
(여러 후행 활동이 있으면 → LS 중 가장 작은 값 - 1)
```

#### 🗺️ 네트워크 예시 (경로 3개)

```
         경로2: A(3일) ──── C(4일)
        /                          \
[Start] ── 경로1: B(5일) ── D(3일) ──[End]   ← Critical Path ⭐
        \                          /
         경로3: E(2일) ──── F(4일)
```

| 경로 | 활동 | 총 기간 | Float | Critical? |
|------|------|---------|-------|-----------|
| 경로 1 | B → D | 5+3 = **8일** | **0** | ⭐ YES |
| 경로 2 | A → C | 3+4 = **7일** | 1 | NO |
| 경로 3 | E → F | 2+4 = **6일** | 2 | NO |

> 📌 Critical Path = 가장 긴 경로 = **8일** = 프로젝트 최단 완료 기간

---

#### ➡️ Step 1: Forward Pass (ES/EF 계산)

> 규칙: `EF = ES + Duration` / 다음 활동 `ES = 이전 EF`

**경로 1 (Critical Path): B → D**

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=5 │──────▶│ ES=5 │ EF=8 │
├──────┴──────┤       ├──────┴──────┤
│   B (5일)   │       │   D (3일)   │
├──────┬──────┤       ├──────┬──────┤
│  ?   │  ?   │       │  ?   │  ?   │
└──────┴──────┘       └──────┴──────┘
```

**경로 2: A → C**

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=3 │──────▶│ ES=3 │ EF=7 │
├──────┴──────┤       ├──────┴──────┤
│   A (3일)   │       │   C (4일)   │
├──────┬──────┤       ├──────┬──────┤
│  ?   │  ?   │       │  ?   │  ?   │
└──────┴──────┘       └──────┴──────┘
```

**경로 3: E → F**

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=2 │──────▶│ ES=2 │ EF=6 │
├──────┴──────┤       ├──────┴──────┤
│   E (2일)   │       │   F (4일)   │
├──────┬──────┤       ├──────┬──────┤

```

#### ⬅️ Step 2: Backward Pass (LS/LF 계산)

> 규칙: `LS = LF - Duration` / 이전 활동 `LF = 다음 LS`
> 프로젝트 종료일 = **8** → 마지막 활동들의 LF = 8

**경로 1 (Critical Path): B → D**

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=5 │──────▶│ ES=5 │ EF=8 │
├──────┴──────┤       ├──────┴──────┤
│   B (5일)   │       │   D (3일)   │
├──────┬──────┤       ├──────┬──────┤
│ LS=0 │ LF=5 │       │ LS=5 │ LF=8 │
└──────┴──────┘       └──────┴──────┘
Float = 0-0 = 0 ⭐      Float = 5-5 = 0 ⭐
```

**경로 2: A → C** ← LF=8에서 역산

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=3 │──────▶│ ES=3 │ EF=7 │
├──────┴──────┤       ├──────┴──────┤
│   A (3일)   │       │   C (4일)   │
├──────┬──────┤       ├──────┬──────┤
│ LS=1 │ LF=4 │       │ LS=4 │ LF=8 │
└──────┴──────┘       └──────┴──────┘
Float = 1-0 = 1         Float = 4-3 = 1
```

**경로 3: E → F** ← LF=8에서 역산

```
┌──────┬──────┐       ┌──────┬──────┐
│ ES=0 │ EF=2 │──────▶│ ES=2 │ EF=6 │
├──────┴──────┤       ├──────┴──────┤
│   E (2일)   │       │   F (4일)   │
├──────┬──────┤       ├──────┬──────┤
│ LS=2 │ LF=4 │       │ LS=4 │ LF=8 │
└──────┴──────┘       └──────┴──────┘
Float = 2-0 = 2         Float = 4-2 = 2
```

#### 📊 최종 결과 요약

| 활동 | Duration | ES | EF | LS | LF | Float | Critical? |
|------|----------|----|----|----|-----|-------|-----------|
| **B** | 5 | 0 | 5 | 0 | 5 | **0** | ⭐ YES |
| **D** | 3 | 5 | 8 | 5 | 8 | **0** | ⭐ YES |
| A | 3 | 0 | 3 | 1 | 4 | 1 | NO |
| C | 4 | 3 | 7 | 4 | 8 | 1 | NO |
| E | 2 | 0 | 2 | 2 | 4 | 2 | NO |
| F | 4 | 2 | 6 | 4 | 8 | 2 | NO |

> 💡 **Float 해석:**
> - B, D → Float=0, 하루라도 늦으면 프로젝트 전체 지연
> - A, C → Float=1, 1일까지는 여유 있음
> - E, F → Float=2, 2일까지는 여유 있음

#### ⚠️ CPM의 핵심 전제 (시험 포인트!)
- ES/EF/LS/LF 계산 시 **자원 제한(Resource Limitation) 고려 안 함**
- 이건 별도 Resource Leveling/Smoothing 단계에서 처리
- **Schedule Baseline은 Early 날짜 기준** (Late 날짜 기준 아님!)
  - Late 날짜로 baseline 잡으면 여유가 0이 돼버림 → 실무상 말이 안 됨

### 3-3-2. Float (여유시간)
- **Total Float:** 전체 프로젝트 지연 없이 활동을 늦출 수 있는 최대 시간
- **Free Float:** 다음 활동 지연 없이 늦출 수 있는 시간

```
Total Float = LS - ES  (또는 LF - EF)
Free Float  = 후행 활동의 ES - 현재 활동의 EF - 1
```

| Float 값 | 의미 |
|----------|------|
| **= 0** | Critical Path 위의 활동. 늦어지면 프로젝트 전체 지연 |
| **> 0** | 여유 있음. 그 시간만큼 늦춰도 OK |
| **< 0** | 이미 일정 초과 상태 → 즉시 대응 필요 |

#### 📝 연습문제 (실전 문제 풀이)

> **Q. Which of the following is NOT a valid statement regarding the critical path method?**
> 
> A. Early/late 날짜 계산 시 자원 제한 고려 안 함  
> B. Forward pass와 Backward pass를 수행함  
> C. Late start/finish 날짜가 일정 기준선(Schedule Baseline)이 됨  
> D. Early/late 날짜의 차이로 일정 유연성(Float)을 측정함

**정답: C** ❌

- A ✅ : CPM은 자원 고려 안 함 (Resource Leveling은 별도)
- B ✅ : Forward(순방향) + Backward(역방향) 패스 수행
- C ❌ : **Baseline은 Early 날짜 기준이지, Late 날짜가 아님**
- D ✅ : Float = LS - ES = 일정 유연성 측정 기준

### 3-3-3. 일정 단축 기법 2가지
|기법|방법|비용|리스크|
|---|---|---|---|
|**Crashing (비용 투입)**|자원 추가 투입해서 단축|증가 ❌|낮음|
|**Fast Tracking (병렬 진행)**|순차 작업을 동시에 진행|변화 없음|증가 ❌|

**시험 포인트:** "일정이 늦어졌는데 예산은 넉넉하다" → **Crashing.** "예산 추가 없이 일정 줄여야 한다" → **Fast Tracking.** 단, Fast Tracking은 재작업 리스크 있음.

### 3-3-4. What-If Scenario Analysis (시나리오 분석)

- **정의:** "X가 발생하면 프로젝트에 어떤 영향이 생기나?"를 사전에 시뮬레이션해서 **대응 계획(Contingency Plan)** 을 만드는 분석 기법
- **목적:** 미래의 불확실한 상황에 대해 미리 대비책 수립
- **언제 쓰나:** 파업, 허가 지연, 공급망 차질 등 예측 가능한 리스크 시나리오 검토 시

> ⚠️ **다른 분석법과 혼동 주의 — 시험 출제 포인트!**

| 분석법 | 목적 | 타이밍 |
|---|---|---|
| **What-If Scenario Analysis** | 미래 시나리오 대비 → **대응 계획 수립** | 사전 (계획 단계) |
| **Variance Analysis** | 계획 vs 실제 차이 분석 (SV, CV 등) | 사후 (모니터링 단계) |
| **Schedule Compression** | 이미 지연된 일정 단축 (Crashing/Fast Tracking) | 사후 (문제 발생 후) |

> "파업, 납품 지연 등에 대비한 대응 계획이 필요하다" → **What-If Scenario Analysis**
>

## 3-4. 원가 관리

### 3-4-0. 비용 유형 4가지 분류

> 💡 **비용 절감 문제 나오면 → Direct + Variable 먼저 봐라**

| 비용 유형 | 한국어 | 의미 | PM 통제 가능? |
|-----------|--------|------|--------------|
| **Direct (직접비)** | 직접비 | 이 프로젝트에만 직접 쓰이는 비용 (인건비, 자재, 전용 장비) | ✅ 높음 |
| **Indirect (간접비)** | 간접비 | 여러 프로젝트가 나눠 쓰는 오버헤드 (임대료, 전기세, 공용 행정직) | ❌ 낮음 |
| **Variable (변동비)** | 변동비 | 작업량에 따라 달라지는 비용 (시간제 인건비, 사용량 기반 자재) | ✅ 높음 |
| **Fixed (고정비)** | 고정비 | 작업량 상관없이 고정 지출 (소프트웨어 라이선스, 장비 리스) | ❌ 낮음 |

**왜 Direct + Variable이 절감 1순위냐:**
- **Direct:** 범위/자원 줄이면 → 직접 투입 비용 바로 줄어듦
- **Variable:** 작업량 줄이면 → 비례해서 같이 줄어듦
- **Indirect:** PM이 컨트롤하는 영역 아님. 범위 줄여도 회사 임대료는 그대로
- **Fixed:** 계약 등으로 묶여 있어서 줄이기 어려움

> ❌ **자주 나오는 함정:** "Indirect + Variable" 선택 → Indirect는 PM이 줄일 수 없음!

### 3-4-1. 산정 기법 4가지:


| 기법                      | 방법                | 정확도   | 시기           |
| ----------------------- | ----------------- | ----- | ------------ |
| **Analogous (유사 산정)**   | 과거 비슷한 프로젝트 기준    | 낮음    | 초반 (정보 적을 때) |
| **Parametric (모수 산정)**  | 단위 비용 × 수량 공식     | 중간    | 중반           |
| **Three-Point (3점 산정)** | 낙관/최빈/비관값 평균      | 높음    | 중반           |
| **Bottom-up (상향식)**     | 작업 단위별 세부 산정 후 합산 | 가장 높음 | 상세 계획 시      |

### 3-4-2. 3점 산정 공식:

```
PERT = (낙관 + 4×최빈 + 비관) / 6
단순 평균 = (낙관 + 최빈 + 비관) / 3
```

### 3-4-3. 예비비 구조

| | Contingency Reserve (우발사태 예비비) | Management Reserve (관리 예비비) |
|---|---|---|
| **대비 대상** | 식별된 리스크 (Known Unknown) | 미식별 리스크 (Unknown Unknown) |
| **Cost Baseline** | ✅ 포함 | ❌ 밖 (Budget에만 포함) |
| **사용 권한** | PM이 직접 사용 가능 | 경영진 승인 필요 |
| **계획 근거** | 리스크 분석 기반 (EMV 등) | 전체 예산의 일정 % |

#### Contingency Reserve 산정 방법 ✅
- **EMV (Expected Monetary Value):** 리스크 발생 확률 × 영향도로 계산
  - 예: 30% 확률로 $10,000 손실 → EMV = $3,000 → 예비비에 반영
- **Reserve Analysis:** 과거 유사 프로젝트 데이터 기반 산정
- **Expert Judgment:** 전문가 판단
- **Monte Carlo Simulation:** 시뮬레이션으로 범위 산출

#### ❌ 틀린 사용 방식 (시험 포인트)
- Contingency Reserve를 **미식별 리스크(Unknown Unknown)** 에 쓰는 것 → Management Reserve 써야 함
- PM이 **Management Reserve를 경영진 승인 없이** 쓰는 것
- 리스크 분석 없이 **그냥 총 예산의 %로만** Contingency 산정하는 것 → 근거 없는 산정은 부적절
- Contingency Reserve 사용 시 **리스크 관리 계획 업데이트 안 하는 것**


## 3-5. 자원 관리 (Resource Management)

### 3-5-0. 자원 관리 프로세스 흐름

```
Plan Resource Management     →    Acquire Resources    →    Develop Team
(자원 관리 계획 수립)               (자원 획득)               (팀 개발)
    ↓                                   ↓
Resource Management Plan          실제 팀원/물적 자원 확보
Staffing Management Plan ⭐       노동 단가 vs 예산 비교
                                  가상 팀 커뮤니케이션 프로토콜 수립
```

> ❌ **핵심 함정:** Staffing Management Plan 작성은 **Planning 단계**. Acquire Resources 단계에서 만드는 게 아님!

### 3-5-1. Acquire Resources (자원 획득) 단계에서 하는 일

| 활동 | 설명 |
|------|------|
| **노동 단가 vs 예산 비교** | 이 자원 고용하면 예산 범위 내인가 체크 |
| **가상 팀 커뮤니케이션 프로토콜 수립** | 원격 팀원 획득 시 소통 방식 정의 |
| **협상 (Negotiation)** | 기능 관리자와 자원 확보 협상 |
| **Pre-assignment 확인** | 사전 배정된 팀원 파악 |

> 📝 **시험 포인트 (TWO 유형 주의!):**
> - "Acquire Resources 단계에서 하는 활동 2개 골라라" → **노동 단가 vs 예산 비교** + **가상 팀 커뮤니케이션 프로토콜**
> - **Staffing Management Plan 작성** → Planning 단계 ❌
> - **Definition of Done / Definition of Ready** → Agile 계획 단계 ❌ (자원 획득과 무관)

### 3-5-2. Virtual Team (가상 팀)

Acquire Resources의 핵심 도구 중 하나. 원격/분산 팀 구성 시 사용.

**장점:** 지리적 제약 없이 전문가 확보, 비용 절감
**단점:** 소통 어려움, 신뢰 구축 시간 필요

> 획득 시 **커뮤니케이션 프로토콜 수립**이 필수 → 어떤 툴로, 얼마나 자주, 어떤 방식으로 소통할지 정의


## 3-6. 변경 통제 (Change Control)

### 3-6-1. 통합 변경 통제 프로세스
- Baseline이 건드려질 때 무조건 변경 통제 프로세스를 거친다.
- Baseline : Scope, Schedule, Cost

```
변경 요청 발생
    ↓
변경 요청서 (Change Request) 작성
    ↓
CCB (Change Control Board) 검토 및 승인/거부
    ↓
승인 시: 계획서 업데이트 + 팀 공지
거부 시: 요청자에게 통보
```

### 3-6-2. CCB (Change Control Board)
- 변경 승인 권한을 가진 공식 기구
- PM 단독으로 중요한 변경 승인 불가

**시험 포인트:**
- "이해관계자가 변경을 요청했다" → 무조건 **변경 요청서 작성 후 CCB 제출.** PM이 임의로 수용/거부 오답.
- "작은 변경이라서 그냥 반영했다" → 오답. 크든 작든 **변경 통제 프로세스 거쳐야 함.**


## 3-7. 점진적 계획 수립 (Rolling Wave Planning)
- 가까운 미래 : 상세하게 계획
- 먼 미래 : 개략적으로만 계획, 가까워지면 상세화
- Agile의 Sprint Planning의 해당 개념의 실천

**시험 포인트:** "초반에 모든 것을 상세하게 계획할 수 없다" → **Rolling Wave Planning.** Agile/Hybrid에서 특히 적합.


# 4. 다른 도메인과의 연결
- **3-1 개발 접근방식:** Predictive는 초반에 상세 계획, Agile은 Rolling Wave
- **3-4 측정:** Cost Baseline이 EVM의 PV 기준이 됨
- **3-6 프로젝트 업무:** 계획대로 실행하고, 편차 발생 시 계획 업데이트
- **3-2 이해관계자:** 이해관계자 참여 계획, 소통 계획이 PMP의 일부


# 5. 시험 포인트
- **"범위가 계속 늘어난다"** → **Scope Creep.** 변경 통제 프로세스 강화.
- **"PM이 고객 만족을 위해 요청 없는 기능을 추가했다"** → **Gold Plating. 오답.** 하면 안 됨.
- **"일정 지연인데 예산 없다"** → **Fast Tracking** (병렬 진행).
- **"일정 지연인데 예산 있다"** → **Crashing** (자원 추가).
- **"프로젝트 초반, 정보 부족한데 산정해야 한다"** → **Analogous 산정.** 가장 빠르고 대략적.
- **"Critical Path 위 활동이 늦어졌다"** → 프로젝트 전체 지연. 즉시 시정 조치.
- **"변경 요청이 왔다, 작은 거라 그냥 해줄까?"** → 오답. **CCB 통해야 함.**
