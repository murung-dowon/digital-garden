>[!tip] 핵심은 Output을 잘 만드는 것보다, 그 Output이 Value로 이어지는지 집중하는 것.

### 1. 개념 정의
- 산출물 (Output) : 프로젝트 활동의 결과로 만들어진 구체적인 '제품'이나 '서비스'
	- 예시 : 새로 만든 맛집 배달앱
- 성과 (Outcome) : 산출물을 통해 나타나는 '변화'나 '결과' 
	- 예시 : 배달 속도 20% 향상, 사용자 만족도 증가 
- 가치 (Value) : 그 성과가 조직이나 고객에게 주는 '실질적인 이득'
	- 예시 : 매출 증대, 시장 점유율 확보 

### 2. 왜 중요한가?
- 기존 (6판)에는 "일정 내에 제품을 잘 만들었는가?" 가 핵심
- 이제는 "제품을 만들어서 진짜 돈이 되었는가? 고객이 좋아하는가?" (Value, Outcome)이 핵심
- 포인트 : "제품은 완벽한데 아무도 안 써서 돈이 안 된다"라고 하면, 그 프로젝트는 **실패**한 것. PM은 항상 '가치'를 모니터링해야 함.

### 3. 실전 예시 (Fitness App)
- **Output:** 운동 기록 기능이 들어간 앱 출시.
- **Outcome:** 사용자들이 꾸준히 운동하게 됨 (건강해짐).
- **Value:** 유료 구독료 수입 발생 + 건강한 사회 구현(공익적 가치).

### 시험 예시
- 산출물은 다 나왔는데 비즈니스 가치가 안보인다면? :
	- 정답 : 비즈니스 케이스를 재검토하고 스폰서와 가치를 재논의해야한다. 
	
- 계획대로 다 만들었는데(Output), 시장이 변해서 아무도 안 쓴대. 어쩔? : 
	- 오답 : "일단 계획된 범위(Scope)는 다 끝냈으니 프로젝트 종료 보고서 쓴다." (이건 6판 마인드)
	- 정답 : 비즈니스 케이스(Business Case) 다시 검토하고, 이 프로젝트 계속하는 게 맞는지 스폰서랑 맞짱(협의) 뜬다.
	- **이유:** 가치(Value)가 없으면 프로젝트는 쓰레기 제조 공정일 뿐

- "고객이 중간에 요구사항을 바꿔달래. 근데 이거 들어주면 원래 만들려던 거(Output)랑 좀 달라짐. 어쩔?"
	- **오답:** "변경 통제 절차(CCB) 빡세게 돌려서 원래 계획대로 밀어붙인다."
	- **정답:** "이 변화가 고객에게 더 큰 성과(Outcome)를 준다면 적극 수용하는 방향으로 검토한다."
	- **이유:** PM은 '문서' 지키는 사람이 아니라 '가치' 만드는 사람임.
