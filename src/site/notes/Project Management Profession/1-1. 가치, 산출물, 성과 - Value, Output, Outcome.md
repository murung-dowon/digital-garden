>[!tip] 핵심은 Output을 잘 만드는 것보다, 그 Output이 Value로 이어지는지 집중하는 것.

## 1. 개념 정의
- 산출물 (Output) : 프로젝트 활동의 결과로 만들어진 구체적인 '제품'이나 '서비스'
	- 예시 : 새로 만든 맛집 배달앱
- 성과 (Outcome) : 산출물을 통해 나타나는 '변화'나 '결과' 
	- 예시 : 배달 속도 20% 향상, 사용자 만족도 증가 
- 이득 (Benefit) : 조직이 실제로 얻는 이득 
	- 예시 : MAU 30% 상승
- 가치 (Value) : 그 성과가 조직이나 고객에게 주는 '실질적인 이득'
	- 예시 : 매출 증대, 시장 점유율 확보 

## 2. 왜 중요한가?
- 기존 (6판)에는 "일정 내에 제품을 잘 만들었는가?" 가 핵심
- 이제는 "제품을 만들어서 진짜 돈이 되었는가? 고객이 좋아하는가?" (Value, Outcome)이 핵심
- 포인트 : "제품은 완벽한데 아무도 안 써서 돈이 안 된다"라고 하면, 그 프로젝트는 **실패**한 것. PM은 항상 '가치'를 모니터링해야 함.

## 3. 실전 예시 (Fitness App)
- **Output:** 운동 기록 기능이 들어간 앱 출시.
- **Outcome:** 사용자들이 꾸준히 운동하게 됨 (건강해짐).
- **Benefit** : 유료 전환율 25% 상승, MAU 40% 증가
- **Value:** 유료 구독료 수입 발생 + 건강한 사회 구현(공익적 가치).

## 4. 가치 인도 컴포넌트

>[!tip] Portfolio > Program > Project > Operations
> 
> **연결 흐름:** Portfolio가 전략 방향 정함 → Program이 관련 Project들 조율 → Project가 결과물 만듦 → Operations가 그 결과물 받아서 계속 돌림

- **Portfolio (포트폴리오):** 조직의 전략 목표 달성을 위해 Program/Project/운영을 묶어서 관리하는 최상위 단위. "우리 회사가 올해 뭘 할 것인가"의 집합.
- **Program (프로그램):** 개별로 관리하면 못 얻는 이득을 위해 관련 프로젝트들을 묶은 단위. "이 프로젝트들이 따로 놀면 시너지가 없으니 묶어서 관리하자"
- **Project (프로젝트):** 고유한 결과물을 만들기 위한 한시적(Temporary) 활동. 시작과 끝이 있음.
- **Operations (운영):** 프로젝트가 끝난 후 결과물이 실제로 돌아가는 반복적·지속적 활동. 프로젝트랑 달리 끝이 없음.

### **실전 예시 (e-커머스 회사):**
- Portfolio: "2026년 디지털 전환 전략" — 이 안에 여러 Program/Project가 있음
- Program: "모바일 고객 경험 개선" — 앱 개발 프로젝트 + 결제 시스템 프로젝트 + 추천 알고리즘 프로젝트를 묶은 것
- Project: "iOS 앱 리뉴얼" — 6개월짜리, 끝나면 해산
- Operations: 앱 출시 후 서버 유지보수, CS 응대, 업데이트 배포 — 매일 반복

### 시험 포인트
- "프로젝트가 끝났는데 PM이 할 일이 끝났나?" → **아니다.** Operations 팀에 결과물 인수인계하고, 그들이 잘 운영할 수 있게 지원하는 것까지가 PM 책임.

- "이 프로젝트 왜 승인됐나?" → Portfolio 전략 목표와 연계되기 때문. 전략이랑 안 맞는 프로젝트는 시작도 안 하는 게 맞음.

- "프로그램이랑 프로젝트 차이가 뭐냐" → 프로그램은 시너지를 위해 묶은 것, 프로젝트는 독립적으로 실행 가능한 단위.

## 5. 정보 흐름 (Information Flow)
**핵심 개념:** 가치 인도 시스템은 정보가 양방향으로 흘러야 제대로 작동함. 한쪽 방향만 있으면 조직이 망함.

**위 → 아래 (전략 전달):**

- Senior Leadership → Portfolio: "우리 올해 전략 목표는 이거야, 이 방향으로 투자해"
- Portfolio → Program/Project: "이 프로젝트가 달성해야 할 성과(Outcome)와 가치(Value)는 이거야"
- Program/Project → Operations: "이 결과물 넘길게, 유지보수 방법은 이렇고 지원 계획은 이거야"

**아래 → 위 (성과 보고):**

- Operations → Program/Project: "현장에서 써보니까 이 부분 고쳐야 할 것 같아" (수정/개선 피드백)
- Program/Project → Portfolio: "현재 진척률은 이렇고, 목표 성과 달성 예상 시점은 이래"
- Portfolio → Senior Leadership: "포트폴리오 전체 실적은 이래, 전략 수정 필요한 부분 있어"

**시험 포인트:**

- "경영진이 프로젝트 현황을 전혀 모른다" → 아래 → 위 정보 흐름이 끊긴 것. PM이 보고 체계 점검해야 함.
- "팀원들이 이 프로젝트가 왜 중요한지 모른다" → 위 → 아래 전략 전달이 안 된 것. PM이 조직 전략과 프로젝트 목표의 연결고리를 팀에 설명해야 함.
- "Operations 팀이 인수받은 시스템 사용법을 몰라 혼란스럽다" → Program/Project → Operations 정보 전달이 부족한 것.

## 6. 시험 예시
- 산출물은 다 나왔는데 비즈니스 가치가 안보인다면? :
	- 정답 : 비즈니스 케이스를 재검토하고 스폰서와 가치를 재논의해야한다. 
	
- 계획대로 다 만들었는데(Output), 시장이 변해서 아무도 안 쓴대. 어쩔? : 
	- 오답 : "일단 계획된 범위(Scope)는 다 끝냈으니 프로젝트 종료 보고서 쓴다." (이건 6판 마인드)
	- 정답 : 비즈니스 케이스(Business Case) 다시 검토하고, 이 프로젝트 계속하는 게 맞는지 스폰서랑 맞짱(협의) 뜬다.
	- **이유:** 가치(Value)가 없으면 프로젝트는 쓰레기 제조 공정일 뿐

- "고객이 중간에 요구사항을 바꿔달래. 근데 이거 들어주면 원래 만들려던 거(Output)랑 좀 달라짐. 어쩔?"
	- **오답:** "변경 통제 절차(CCB) 빡세게 돌려서 원래 계획대로 밀어붙인다."
	- **정답:** "이 변화가 고객에게 더 큰 성과(Outcome)를 준다면 적극 수용하는 방향으로 검토한다."
	- **이유:** PM은 '문서' 지키는 사람이 아니라 '가치' 만드는 사람임.
