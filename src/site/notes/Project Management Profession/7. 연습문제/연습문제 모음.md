## PMP 문제 마인드셋
### 오답 패턴 — 이 단어 보이면 일단 걸러
❌ Confront     → 대립적
❌ Demand       → 강압적
❌ Strict orders → 명령
❌ Force        → 강제
❌ Wait         → 방치
❌ Ignore/Disregard → 무시
❌ Minimal support → 지원 없이 던짐
❌ Competitive environment → 팀 동기부여에 경쟁 조장
❌ Convince (someone to stay) → PM 통제 밖

### PMI 마인드셋 — 애매할 때 역추적 원칙

**원칙 1: 에스컬레이션은 항상 마지막**

```
PM이 직접 해결 가능 → 스폰서/PMO에 먼저 가면 ❌
진짜 PM 권한 밖일 때만 → Escalate
```

**원칙 2: Engagement > Communication**

```
정보 전달(보고서, 이메일) < 직접 참여/대화
"send a report" vs "schedule a meeting" → 미팅이 거의 정답
```

**원칙 3: 예방 > 감지 > 수정**

```
"avoid this situation" 문제 → 가장 근본적 예방책 고르기
분석/모니터링보다 구조적 예방이 더 높은 점수
```

**원칙 4: 분석 먼저 vs 행동 먼저**

```
상황이 불명확할 때 → 분석/검토 먼저
이미 정보 있고 문제 명확할 때 → 행동 먼저
충돌/갈등 → 바로 Facilitate meeting
```

**원칙 5: "Do FIRST" 문제 패턴**

```
새 상황 발생 → 분석/검토 먼저
이미 분석된 상황 → 액션
승인 완료 → 문서 업데이트
```

**원칙 6: 두 선지가 다 맞아 보일 때**

```
더 포괄적인 것 (C vs A라면 C가 A를 포함하면 C)
더 근본적인 것 (증상 해결 vs 원인 해결 → 원인)
더 직접적인 것 (PM이 직접 하는 것 > 위임/에스컬레이션)
```

**원칙 7: 이해관계자 관련**

```
저항/갈등 → Early Engagement
새 이해관계자 → 즉시 직접 미팅
참여 저조 → 의사결정 과정에 포함
Low Power, High Interest → Keep Informed (A 수준)
High Power, High Interest → Manage Closely
```

**원칙 8: 팀 관련**

```
항상 Theory Y (자율, 신뢰)
갈등 → Collaborate 먼저
새 방법론 저항 → 교육, 전략 바꾸기 ❌
```


## PMP 공식 문제은행
### Mentor and Collaborate With Stakeholders
- Stakeholder Register = "누가 있고 어떤 사람이냐" (목록)
- Stakeholder Engagement Plan = "각자에게 어떻게 소통할거냐" (전략)
- Communications Mgmt Plan    = "전체 소통을 어떻게 운영할거냐" (전반적 방식)

---

- 메가프로젝트 : 팀원들끼리 상호 소통 > 1대1 소통 
- PM은 단순 정보 전달보다 항상 쌍방향 Collab을 지향해야함. 
	- 효율보다 engagement가 중요하다. (적확한 정보 전달)
	  
- Power / Interest Grid 개념 잘 숙지하기.
- 에자일 프로젝트는 모든 스프린트 과정에 이해관계자가 포함되어야함.
	- (분석) 리스크 / 문제 발생 -> 분석 먼저 
	- (참여) 이해관계자 참여 -> 원래 해야하는 것
	
- 충돌 문제 나오면 Collaborate - 미팅으로 같이 해결해야함. 
- PMP는 올바른 액션을 취했냐를 봄 - 행동의 적절성 기준
- PM이 리소스 품질에 대해 책임진다 
- PMP는 말보다 문서로 해결하는 것을 선호

### Plan and Manage Project/phase Closure or Transitions
- 성공 기준(success criteria)은 프로젝트 시작 때 **Project Charter에 공식 정의**
- 문제 발생 시 **무조건 분석 먼저** — PMP 황금 원칙

### Manage Project Issues
- 문제 발견 → 무조건 영향 분석 먼저
- Agile에서 "진행 상황 혼란" → Iteration Board (Scrum/Kanban Board)
- 종료된 프로젝트 관련 문제 발생 → PMIS 아카이브 (과거 기록 조회)
- Functional Manager : 부서장, 팀장 (디자인 팀장 등)

### Plan and Manage Scope
- "요구사항이 계속 변한다" = Agile 맥락 → Product Owner와 지속적 소통 + 점진적 승인이 핵심
- WBS는 Scope 정의에서 시작
	- 프로젝트 헌장 -> 스코프 정의 -> WBS 생성 -> 요소 상세 정의 -> 액티비티 리스트
- Customer Focused는 WBS 구조 특성 아님

### Address and Remove Impediments, Obstacles,
- PMP에서의 Roadmap = 언제? (실행 타임라인)
- PM의 역할 = 팀원이 스스로 우선순위 판단할 수 있게 **지원**하는 것

### Manage Conflict
- 개선/프로세스 문제 = 무조건 Retrospective!
- PMP에서 갈등 = 불가피한 것. 피하는 게 아니라 관리하는 것.
- PM이 모든 갈등 해결의 중심이 되면 안 됨 → 팀 자율성 키워야 함
- "훈련/교육이 필요하다" → 무조건 Needs Assessment 먼저 → Plan은 그 다음
- PMP는 **비난 금지**지 **대화 금지**가 아님 ㅋㅋ "Speak to" = 건설적 대화 → PMP에서 권장하는 방식 맞음.
- Resource management plan자원 관리 방법론 정의. 개별 업무 배정 매핑 아님

### Build and Lead a Team

팀 역량 낮음 → High directive
팀 역량 높음 → Low directive
팀 동기/관계 필요 → High supportive
팀 자립적 → Low supportive

Internal + High power  → 경영진, 스폰서, PMO
External + High power  → 규제기관, 주요 고객 ← 이번 문제
Internal + Low power   → 일반 팀원
External + Low power   → 일반 공급업체, 일반 대중

Scrum 팀 협업 문제
→ 통제/지시 강화 ❌ (A, B, C)
→ 비전 + 임파워먼트 + 책임감 ✅ (D)

자율적 팀 = 비전이 명확할 때 자연스럽게 협업함

**① Directive (지시형)**

```
PM이 직접 지시, 통제
언제: 팀 역량 낮음 / 위기 상황
특징: 빠른 결정, 경직됨
```

**② Collaborative (협업형)**

```
팀 함께 결정, 참여 중심
언제: 변화/불확실 환경
특징: 동기↑, 혁신↑, 유연함
```

**③ Servant Leadership (서번트)**

```
PM이 팀을 섬김
언제: Agile/Scrum 환경
특징: 장애물 제거, 팀 우선
```

**④ Transformational (변혁적)**

```
비전으로 팀 변화/성장 이끔
언제: 조직 변화, 혁신 필요
특징: 영감, 동기부여, 장기적
```

**⑤ Transactional (거래적)**

```
보상/처벌로 성과 관리
언제: 단기 목표, 명확한 KPI
특징: if-then 구조 (Expectancy Theory랑 연결)
```
