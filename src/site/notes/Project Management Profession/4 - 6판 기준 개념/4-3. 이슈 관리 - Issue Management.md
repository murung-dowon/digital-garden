> [!tip] "리스크는 미래, 이슈는 현재. 이미 터진 문제는 이슈로 관리해라."

# 1. 한 줄 요약

- **이슈(Issue)** = 이미 발생한 문제. 지금 당장 해결해야 하는 것.
- 리스크와 구분 필수. 리스크가 현실화되면 이슈가 됨.

---

# 2. 리스크 vs 이슈 — 핵심 구분

| | **Risk (리스크)** | **Issue (이슈)** |
|---|---|---|
| **발생 여부** | 아직 안 일어남 (미래) | 이미 발생함 (현재) |
| **관리 도구** | Risk Register | **Issue Log** |
| **대응 방식** | 예방·완화·회피·전가 | 즉각 해결, 담당자 배정 |
| **예시** | "공급사 납품 지연될 수도 있다" | "공급사가 납품을 못 하겠다고 통보했다" |

> 📝 **시험 포인트:** "리스크가 현실화됐다" → 이제 **이슈**로 관리. Risk Register에서 Issue Log로 이동.

---

# 3. Issue Log (이슈 로그)

이슈를 추적·관리하는 공식 문서. 포함 항목:

| 항목 | 설명 |
|------|------|
| **이슈 ID** | 고유 식별번호 |
| **이슈 설명** | 무슨 문제인지 |
| **발견일** | 언제 발생했는지 |
| **우선순위** | High / Medium / Low |
| **담당자 (Owner)** | 누가 해결 책임지는지 |
| **목표 해결일** | 언제까지 해결해야 하는지 |
| **상태** | Open / In Progress / Closed |
| **해결 방법** | 어떻게 해결했는지 |

> 📝 **시험 포인트:** "팀원 간 의견 충돌이 반복된다" → Issue Log에 기록하고 담당자 배정. "그냥 넘어가자" → 오답.

---

# 4. 이슈 처리 프로세스

```
이슈 발생
    ↓
Issue Log에 기록 (ID, 설명, 우선순위)
    ↓
담당자(Owner) 배정
    ↓
해결 방안 수립 및 실행
    ↓
해결 확인 → Issue Log 업데이트 (Closed)
    ↓
필요 시 Change Request 생성 (범위/일정/비용 영향 있을 경우)
```

> ⚠️ **이슈가 Baseline에 영향주면** → Change Request → CCB 승인 필요. 이슈 해결이라고 임의로 범위/일정 바꾸면 안 됨.

---

# 5. 에스컬레이션 (Escalation)

PM 선에서 해결 안 되는 이슈 → 상위 레벨로 올려야 함.

| 상황 | 에스컬레이션 대상 |
|------|-----------------|
| 자원/예산 권한 밖의 이슈 | 스폰서 |
| 조직 간 갈등 | 프로그램 매니저 / PMO |
| 계약 관련 이슈 | 법무팀 / 조달팀 |

> 📝 **시험 포인트:** "PM이 혼자 해결할 수 없는 이슈" → **에스컬레이션**. 무조건 혼자 끌어안으려 하면 오답. 단, 에스컬레이션 전에 먼저 본인이 해결 시도하는 게 맞음.

---

# 6. 시험 포인트

- **"리스크가 발생했다"** → 이제 **이슈**. Issue Log에 기록.
- **"팀원이 계속 같은 문제를 일으킨다"** → Issue Log에 기록 + 담당자 배정 + 해결 추적.
- **"이슈 해결하려면 범위를 바꿔야 한다"** → **Change Request** 제출 → CCB 승인.
- **"PM이 해결 못 하는 이슈"** → 에스컬레이션. 혼자 묻어두면 오답.
- **"이슈 관리 안 하면?"** → 작은 이슈가 프로젝트 전체 위기로 커짐. PM은 이슈를 가시화(Visible)해야 함.
