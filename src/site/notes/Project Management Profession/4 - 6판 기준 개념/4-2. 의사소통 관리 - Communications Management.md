# 1. 한 줄 요약

- 누가, 언제, 무엇을, 어떤 방식으로 받아야 하는지 계획하고 실행하는 것.
- 프로젝트 실패의 주요 원인 중 하나가 의사소통 문제. PM 시간의 약 **90%**가 소통에 쓰임.

---

# 2. 의사소통 관리 프로세스

```
Plan Communications Management   →   Manage Communications   →   Monitor Communications
(의사소통 계획 수립)                    (의사소통 실행)               (의사소통 모니터링)
        ↓
Communication Management Plan
```

---

# 3. Communication Management Plan (의사소통 관리 계획서)

"누가 뭘 언제 어떻게 받는가"를 정의한 문서. 포함 내용:

| 항목 | 예시 |
|------|------|
| **수신자 (Who)** | 스폰서, 팀원, 이해관계자 |
| **내용 (What)** | 주간 상태 보고, 리스크 현황, 예산 현황 |
| **주기 (When)** | 매주 금요일, 마일스톤 완료 시 |
| **방법 (How)** | 이메일, 회의, 대시보드 |
| **담당자 (Who sends)** | PM, 팀 리더 |

> 📝 **시험 포인트:** "이해관계자가 정보를 못 받았다고 불평한다" → Communication Management Plan 재검토 및 업데이트.

---

# 4. 의사소통 채널 수 공식

> 이해관계자가 늘어날수록 소통 복잡도가 기하급수적으로 증가함.

```
채널 수 = n(n-1) / 2
(n = 이해관계자 수)
```

| 이해관계자 수 | 채널 수 |
|-------------|--------|
| 2명 | 1개 |
| 5명 | 10개 |
| 10명 | 45개 |
| 20명 | 190개 |

> 📝 **시험 포인트:** "팀원이 5명 추가됐다. 채널은 몇 개 늘었나?" → 추가 전후 채널 수 차이 계산.

---

# 5. 의사소통 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **Interactive (양방향)** | 실시간 쌍방 소통. 즉각 피드백 가능 | 회의, 전화, 화상통화 |
| **Push (일방 발신)** | 발신자가 수신자에게 전달. 확인 여부 모름 | 이메일, 보고서, 공문 |
| **Pull (수신자 선택)** | 수신자가 필요할 때 꺼내 봄 | 인트라넷, 공유 드라이브, 위키 |

> 📝 **시험 포인트:**
> - "복잡한 이슈를 논의해야 한다" → **Interactive** (회의)
> - "많은 사람에게 공지사항 전달" → **Push** (이메일)
> - "팀원들이 필요할 때 참조할 문서" → **Pull** (공유 드라이브)

---

# 6. 가상 팀 (Virtual Team) 의사소통

원격/분산 팀 소통 시 추가로 고려할 것:

- 시간대 차이 → 회의 시간 조율
- 문화적 차이 → 소통 방식 명확히 정의
- 커뮤니케이션 프로토콜 문서화 필수
- 화상회의, 협업 툴(Slack, Teams 등) 활용

→ [[Project Management Profession/3 - 8 Project Performance Domains/3-5. 계획\|3-5. 계획]] 자원 관리 섹션과 연결

# 7. 시험 포인트

- **"PM이 시간을 가장 많이 쓰는 활동"** → **소통(Communication)**. 약 90%.
- **"이해관계자가 정보 공유 안 된다고 불만"** → Communication Plan 업데이트 + 소통 채널 점검.
- **"팀원이 추가됐다"** → 채널 수 재계산. 소통 복잡도 급증 인식.
- **"민감한 정보 전달"** → Interactive (직접 대화). 이메일(Push) 단독 사용 부적절.
- **"기록으로 남겨야 하는 소통"** → Push (이메일, 공문). 구두 전달만 하면 오답.
