# 시험 포인트 모음 (문제 풀다 발견한 것들

---

## ✅ Project Management Plan 구조
> 관련 필기: [[Project Management Profession/3 - 8 Project Performance Domains/3-5. 계획\|3-5. 계획]] → 3-1 섹션

- PMP = **최상위 통합 문서**, 모든 하위(Subsidiary) 계획을 포함
- HR Management Plan, Schedule Management Plan 등 **~~ Plan** 이름이면 PMP 안에 포함

**훼이크 주의 - Plan vs Register/Log:**

| PMP에 포함 ✅ | PMP에 미포함 ❌ |
|---|---|
| Risk Management Plan | Risk Register |
| Stakeholder Engagement Plan | Stakeholder Register |
| Quality Management Plan | Quality Control Measurements |
| (모든 ~~ Plan) | Issue Log, Change Log |

> **Rule of thumb:** `~~ Plan` → PMP 포함 / `~~ Register`, `~~ Log` → Project Document (별도)

---

## ✅ Project File vs Project Deliverable
> 관련 필기: [[Project Management Profession/3 - 8 Project Performance Domains/3-6. 프로젝트 업무\|3-6. 프로젝트 업무]]

- **Project File** = 프로젝트를 *관리*하면서 생성된 문서/기록
  - Change management documentation, Project calendars, Project management plan 등
- **Project Deliverable** = 프로젝트가 *만들어낸 결과물* 자체 → **Project File 아님**
  - 고객한테 넘기는 소프트웨어, 보고서, 건물 등

**시험 포인트:** Lessons Learned 작성 시 검토하는 건 "어떻게 관리했냐"의 기록(Project File)이지, 산출물(Deliverable) 자체가 아님.

## ✅ Conduct Procurements vs Procurement Management
> 관련 필기: 없음 (PMBOK 6판 내용)

- **Procurement Management** = 조달 관리 전체 Knowledge Area (계획~통제 다 포함하는 큰 틀)
- **Conduct Procurements** = 그 안의 특정 실행 프로세스 → **벤더 선정 + 계약 실행이 여기서 발생**

**시험 포인트:** 둘 다 선택지에 나오면 **더 구체적인 게 답**. "Vendor selection"처럼 실행 행위가 언급되면 Conduct Procurements.

Lessons Learned는 프로젝트 종료 때 한꺼번에 하는 게 아니라 **각 프로세스 진행하면서 계속 업데이트** → Conduct Procurements 완료 시에도 기록함.

## ✅ PDCA → PMBOK Process Group 매핑
> 관련 필기: 없음 (PMBOK 6판 내용)

| PDCA | PMBOK Process Group |
|---|---|
| **Plan** | Planning |
| **Do** | Executing |
| **Check** | Monitoring & Controlling |
| **Act** | Monitoring & Controlling ← 헷갈림 주의! |

**왜 Act가 Executing이 아니고 M&C인가?**
- Do = 그냥 계획대로 실행
- Act = 체크 결과 바탕으로 **시정 조치(Corrective Action) 취하는 것**
- M&C는 단순히 "모니터링만 하는" 프로세스가 아니라, 편차 발견 후 **시정 조치 실행까지 포함**
- 그래서 Check + Act 둘 다 M&C에 해당

## ✅ Risk Transference (리스크 전가) - 계약 유형별 구분
> 관련 필기: [[Project Management Profession/2 - 12 Principles of Project Management/2-10. 리스크 - Risk\|2-10. 리스크 - Risk]] → Transfer(전가) 항목

**Transfer = 리스크를 제3자에게 넘기는 것. 리스크가 사라지는 게 아니라 책임 주체가 바뀌는 것.**

| 계약/수단 | 한국어 | Transfer 해당 여부 | 이유 |
|---|---|---|---|
| **Fixed Price** | 고정 가격 계약 | ✅ 전가 맞음 | 비용 초과 리스크를 계약업체가 부담 |
| **Warranties** | 보증 | ✅ 전가 맞음 | 결함 책임을 공급업체한테 넘김 |
| **Performance bonds** | 이행 보증 | ✅ 전가 맞음 | 계약 불이행 리스크를 보증사(제3자)에게 넘김 |
| **Cost Reimbursable** | 실비 정산 계약 | ❌ 전가 아님 | 실제 발생 비용을 발주자가 전액 부담 → 리스크가 내 쪽에 있음 |

**Warranty vs Performance Bond 차이:**
- **Warranty (보증)** = "내가 만든 거 결함 있으면 내가 책임질게" → 공급업체한테 전가
- **Performance Bond (이행 보증)** = "이 업체가 계약 못 이행하면 내(보증사)가 물어줄게" → 제3자 보증사한테 전가

**시험 포인트:** Cost Reimbursable은 Transfer처럼 들리지만 오히려 반대. 비용 리스크를 **발주자(나)가 떠안는** 구조라 전가 아님.
