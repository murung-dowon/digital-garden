# PMP Official Study Log

---

## ⚡ 핵심 체크리스트 — 틀렸던 개념 위주

### 리스크 관리
- `approved actions 완료` → Risk Mgmt Plan 업데이트 ❌ → **Monitor Risks 진입**
- `probability & impact 업데이트` 참고 문서 → Marketplace conditions ❌ → **Risk Register**
- `"may delay" / "might arise" / "could affect"` → Issue Log ❌ → **Risk Register** / 확정되면 그때 Issue Log
- ⚠️ 문제에 "issue"라는 단어 있어도 **might/could/may 붙으면 Risk Register**
- `기회 발생` → 승인 요청 먼저 ❌ → **리스크 분석 먼저**, 승인은 나중
- `대응 결정 후 PM Plan 업데이트 근거` → Lessons Learned ❌ → **Risk Register**
- `Monitor Risks 도구` → Affinity grouping ❌ (Identify 단계) → **Data analysis + Risk audits**
- `DoD 없음` → Management Risk ❌ → **Technical Risk** (What=기술 / How=관리)
- `이미 알려진 리스크 발생` → 경영진 재고 건의 ❌ → **초기 계획 먼저 확인**
- `Risk Attitude` → **averse / neutral / tolerant / seeking** (Conservative 같은 거 아님)
- `미식별 돌발 상황 비용 초과` → Contingency Reserve ❌ → **Management Reserve** (반대로 외우기 쉬움 주의!)

### 리더십 & 팀
- `"behavioral orientation"` → Directive ≠ Control → **Directive = 명확한 커뮤니케이션**
- `Competitive environment` → 팀 동기부여 방법 ❌ → **PMP는 항상 협업 선호**
- `Predictive → Hybrid 전환` → Scrum/Lean 방법론 ❌ → **Servant Leadership**
- `if-then 보상 구조` → Achievement Theory ❌ → **Expectancy Theory (Vroom)**
- `변화/불확실 환경` → Directive ❌ → **Collaborative** (동기/혁신/유연성)
- `자율/신뢰/감독 최소화` → Theory X ❌ → **Theory Y**
- `팀원 문제 행동` → 공개 질책/즉시 징계 ❌ → **1:1 개인 면담 먼저**
- `"insist" + 개인 면담` → 강압적 ❌ → **명확한 기대치 전달 (리더십)**

### 리더십 스타일 6가지
| 스타일 | 언제 | 특징 |
|--------|------|------|
| **Directive** | 팀 역량 낮음 / 위기 | 직접 지시, 통제 |
| **Collaborative** | 변화/불확실 환경 | 함께 결정, 동기↑ |
| **Servant** | Agile/Scrum | 팀 섬김, 장애물 제거 |
| **Transformational** | 조직 변화/혁신 | 비전으로 영감, 장기적 |
| **Transactional** | 단기 목표/KPI | 보상/처벌, if-then |
| **Laissez-faire** | 고숙련 자율팀 | 최소 개입, 위임 |

### Situational Leadership (Hersey-Blanchard)
```
S1. High directive + Low supportive  → 신입/초보팀
S2. High directive + High supportive → 개발 중인 혼합팀
S3. Low directive + High supportive  → 숙련but자신감↓
S4. Low directive + Low supportive   → 베테랑 자율팀
```

### 이해관계자 관리
- `저항 예방` → Communication 투자 / Agreement ❌ → **Early Engagement** (함께 결정)
- `다국가 프로젝트` → 관리자에게 예외 요청 ❌ → **PM이 직접 문화 사전 평가**

### 변경 관리
- `"approved"` 보이면 → 승인 단계 이미 끝 → **계획 문서 업데이트**

# 드래그앤드롭 대비 핵심 순서/프로세스 플로우

### ① 리스크 관리 7단계
```
1. Plan Risk Management → 2. Identify Risks → 3. Qualitative Analysis
→ 4. Quantitative Analysis → 5. Plan Responses → 6. Implement Responses → 7. Monitor Risks
```

### ② 변경 통제 프로세스
```
변경 요청 → 영향도 분석 → CCB 승인 → PM Plan 업데이트 → 커뮤니케이션
```

### ③ Tuckman 팀 개발 5단계
```
Forming(방향) → Storming(갈등) → Norming(지원) → Performing(위임) → Adjourning(마무리)
```

### ④ 갈등 해결 강도 순서 (약→강)
```
Withdraw → Smooth → Compromise → Force → Collaborate ✅ (PMP 권장)
```

### ⑤ Agile Sprint 사이클
```
Sprint Planning → Daily Standup → Sprint 실행 → Sprint Review → Sprint Retrospective
```

### ⑥ 조달 프로세스
```
Plan → Conduct (RFI→RFP→RFQ) → Control → Close
```

### ⑦ 이슈 에스컬레이션 순서
```
PM 직접 → 팀 내 → Functional Manager → Sponsor → 최고경영진
```

### ⑧ WBS → 일정 수립 순서
```
WBS → Activity List → 순서 배열 → 자원 산정 → 기간 산정 → 일정 개발 → 일정 통제
```

### 품질 관리
- `외부 감사자/컨설턴트 + 방법론 검토` → Control Quality ❌ → **Manage Quality**
- `테스트/결함/검수` → Manage Quality ❌ → **Control Quality**
- `일정 정상인데 품질 희생 제안` → 논의/수락 ❌ → **즉각 거절** (정당한 이유 없는 품질 타협)
- `"요구사항 완료 측정 방법"` → Quality Mgmt Plan ❌ → **Requirements Management Plan**

### 조달 관리
- `납품물이 인수 기준 미달 → 벤더 검증 근거` → RFP/RFQ ❌ → **SOW** (합의된 인수 기준이 여기 있음)
- `신규 벤더에게 줄 가장 포괄적인 문서` → RFI/RFQ ❌ → **RFP** (요구사항 + 인수 기준 전부 포함)
- `key deliverables를 묘사하는 문서` → Agreements ❌ → **Project Scope Statement** (Agreement = 계약 조건, Scope Statement = deliverables 정의)
- 정보량: **RFP > RFC ≈ RFQ > RFI**

### Agile
- `User Story 모호해서 생긴 문제` → end users에게 요구사항 수집(D) ❌ → **User Story를 더 상세하게 작성**
- `팀원 행동 규칙 위반` → Team-building ❌ → **기존 Ground Rules 집행**

### PMIS / 정보 관리
- `과거 프로젝트 이력 + 리소스 + 성과 데이터 종합 조회` → 헌장/품질도구/보고서 ❌ → **PMIS** (모든 프로젝트 정보의 중앙 허브)
- PMIS는 부분집합(보고서, 헌장 등)이 아닌 **전체를 포함하는 시스템**이 포인트

# 리스크 관리 (Risk Management)

### 리스크 관리 7단계 프로세스

> 핵심: 어떤 문서가 업데이트되냐 = 지금 어느 단계냐

#### 전체 흐름
```
1. Plan Risk Management       → 리스크 관리 방법 계획
         ↓
2. Identify Risks             → 리스크 목록 식별
         ↓
3. Qualitative Risk Analysis  → 확률/영향도 평가, 우선순위 결정
         ↓
4. Quantitative Risk Analysis → 수치로 리스크 영향 분석 (선택적)
         ↓
5. Plan Risk Responses        → 대응 전략 수립
         ↓
6. Implement Risk Responses   → 대응 계획 실행
         ↓
7. Monitor Risks              → 지속적 모니터링, 새 리스크 식별
```

#### 단계별 업데이트 문서

| 단계                                 | 업데이트 문서                                                                                                |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **1. Plan Risk Management**        | Risk Management Plan (이후 업데이트 거의 없음)                                                                   |
| **2. Identify Risks**              | **Risk Register 최초 생성, Risk Report 최초 생성**, Project Documents Updates (Assumption Log, Issue Log 등 포함) |
| **3. Qualitative Risk Analysis** ⭐ | Risk Register, Risk Report, Assumption Log, **Issue Log** (4개 전부)                                      |
| **4. Quantitative Risk Analysis**  | Risk Register, Risk Report                                                                             |
| **5. Plan Risk Responses**         | Risk Register, Risk Report, Change Requests                                                            |
| **6. Implement Risk Responses**    | Issue Log, Change Requests                                                                             |
| **7. Monitor Risks**               | Risk Register, Issue Log, **Lessons Learned** ← 여기서만 나옴                                                |

#### 시험 포인트
- **Qualitative 끝난 후** → 4개 무조건 (Register, Report, Assumption Log, Issue Log)
- **Lessons Learned** → Monitor Risks(7단계)에서만. 앞 단계 선지에 나오면 ❌ 오답
- **Plan Risk Management** → 1단계 딱 한 번. 이후 선지에 나오면 ❌ 오답
- **Issue Log** → Qualitative(3단계)부터 시작. "분석하다 이슈 발견하면 바로 기록"

---

### Qualitative vs Quantitative 리스크 분석

> 핵심 키워드: **"prioritize + probability & impact"** = Qualitative / **"수치·시뮬레이션"** = Quantitative

#### 정의 비교

| | Qualitative (3단계) | Quantitative (4단계) |
|---|---|---|
| **목적** | 리스크 **우선순위** 결정 | 리스크의 **수치적 영향** 분석 |
| **방법** | High/Med/Low, P×I 매트릭스 | 몬테카를로, 결정트리, 민감도 분석 |
| **질문** | "이 리스크 얼마나 심각해?" | "이 리스크로 얼마나 손해봐?" |
| **필수 여부** | ✅ 항상 수행 | ❌ 선택적 (대형 프로젝트) |

#### 킬러 키워드
- **Qualitative** → prioritize, probability of occurrence, impact, P×I matrix, further analysis or action
- **Quantitative** → numerical analysis, monetary value, Monte Carlo simulation, EMV, decision tree, sensitivity analysis

#### Quantitative 대표 도구
- **Monte Carlo Simulation**: 수천 번 시뮬레이션 → 확률 분포 산출
- **Decision Tree + EMV**: 기대화폐가치(Expected Monetary Value) 계산
- **Sensitivity Analysis**: 어떤 리스크가 결과에 가장 큰 영향을 주는지 파악

> ⚠️ Quantitative는 **반드시 수학적/통계적 모델** 사용. 숫자 없으면 Quantitative 아님.

### 리스크 대응 완료 후 → Monitor Risks 전환

> 핵심: **"approved actions 완료"** = Implement Risk Responses(6단계) 종료 → 다음은 **Monitor Risks(7단계)**

#### 문제 패턴
> "A large risk is identified, risk management plan is executed. Upon completion of approved actions, what should PM do?"
> → 정답: **D. Identify new risks, and monitor residual risks**

```
리스크 식별 → 분석 → 대응계획 수립 → 대응 실행(Approved Actions) ✅ 완료
                                                    ↓
                                         Monitor Risks (7단계)
                                         = 새 리스크 식별 + 잔여 리스크 모니터링
```

#### 시험 포인트
- **"approved actions 완료"** 보이면 → 이미 6단계(Implement) 끝 → 7단계(Monitor)로 이동
- **Residual risk** = 대응 후에도 남아있는 리스크 → 계속 모니터링 필요
- **Secondary risk** = 대응 조치 자체가 만들어낸 새 리스크 → 역시 Identify & Monitor

### Risk Register — probability & impact 참고 문서 (⚠️ 문장 구조 트릭)

> 핵심: **"probability and impact 검토/업데이트"** 키워드 → **Risk Register**

#### 문제 패턴
> 환율 변동 리스크, PM이 현재 금융 상황으로 probability & impact 업데이트하려 함 → 뭘 참고?
> → 정답: **B. Risk Register**

#### 문장 구조 트릭
```
"update the probability and impact of a new scenario
 using current financial conditions"
       ↑                                    ↑
  뭘 업데이트?                          어떤 데이터로? (수단)
  prob & impact                        Marketplace conditions
  → 기록된 문서 = Risk Register        → 이건 외부 인풋, "refer to" 대상 아님
```

> ⚠️ "using ~" = 업데이트 방법을 수식하는 부사구. 질문은 "뭘 참고하냐"이지 "어떤 데이터를 쓰냐"가 아님

#### 킬러 키워드 → 문서 매핑

| 키워드 | 참고/업데이트 문서 |
|---|---|
| probability & impact 검토/업데이트 | **Risk Register** |
| approved actions 완료 후 다음 단계 | **Monitor Risks** 진입 |
| change request approved | **Project Management Plan 업데이트** |

### MAY delay vs IS delayed — Risk vs Issue

> 핵심: **"may"** = 아직 가능성 = Risk Register / **"has/is"** = 이미 발생 = Issue Log

#### 상황 1: "MAY delay" (가능성)

```
공급업체: "납품 늦을 수도 있어요"
                    ↓
           불확실한 미래 사건 = RISK
                    ↓
        Risk Register 업데이트
        ├── Probability & Impact 평가
        └── 대응 전략 수립 (Mitigate / Transfer / Accept)
```

#### 상황 2: "IS delayed" (확정)

```
공급업체: "납품 2주 늦어집니다"
                    ↓
           이미 발생한 문제 = ISSUE
                    ↓
        Issue Log 업데이트
        └── 파생 업데이트
             ├── Risk Register: 해당 리스크 "발생함"으로 상태 변경
             ├── Project Schedule: 영향 일정 조정
             └── Change Request: 필요시

```

> ⚠️ **Risk → Issue는 단방향**. Issue가 되면 Risk Register도 같이 건드려야 함.

| 키워드 | 문서 |
|---|---|
| may, might, could, potential | **Risk Register** |
| has, is, occurred, confirmed | **Issue Log** |

### 새로운 기회 발생 시 — 분석이 승인보다 먼저

> 핵심: **리스크 분석 → 승인 요청** 순서. 분석 없이 승인 요청하러 가면 ❌

#### 문제 패턴
> 장비 조기 납품 가능 → PM이 기회 인식 → 다음 액션?
> → 정답: **D. Determine if there is a risk** (A. Request management approval 아님!)

```
기회/위협 인식 → 리스크 분석 → (필요시) 승인 요청 → 일정 수정 → 실행
```

> ⚠️ **"What should PM do NEXT?"** + 새 상황 발생 → 무조건 **분석 먼저**
> Opportunity도 Risk — 조기 납품도 보관비용, 현금흐름, 순서 충돌 등 리스크 있을 수 있음

### 리스크 카테고리 4가지

| 카테고리 | 핵심 | 예시 |
|---|---|---|
| **Technical** | **"무엇을 만드냐(What)"** 불명확 — 범위, 요구사항, 기술 기준, DoD | DoD 없음, 요구사항 불명확, 기술 검증 실패 |
| **Management** | **"어떻게 운영하냐(How)"** 불명확 — 자원, 조직, 커뮤니케이션 | 자원 부족, 역할 불명확, 소통 계획 없음 |
| **External** | 조직 외부 요인 | 법규 변경, 경쟁사, 환경 규제 |
| **Commercial** | 거래/계약 관련 | 조달 계약, 하청업체, 파트너십 |

> ⚠️ **DoD 없음 = Technical** (관리 문제처럼 보여도 "완성 기준" = 제품/요구사항 영역)
> Technical vs Management 헷갈리면 → What(제품) vs How(운영) 으로 판단

### Monitor Risks 도구/기법

| 도구 | 설명 |
|---|---|
| **Data analysis** ✅ | 추세·분산 분석으로 리스크 현황 파악. 터지기 전에 신호 포착 |
| **Risk audits** ✅ | 대응 전략이 효과적으로 작동하는지 주기적 점검 |
| **Meetings** | 리스크 리뷰 미팅 |

> ⚠️ Affinity grouping → Identify 단계 / Risk Transference → Plan Response 단계
> 모니터링 = 단순 관찰이 아니라 **"대응이 효과 있나?" 데이터로 검증**하는 것

### 리스크 대응 전략 5가지

#### Threat (부정적 리스크) 대응

| 전략 | 한 줄 설명 | 비유 | 핵심 |
|---|---|---|---|
| **Avoid (회피)** | 리스크 자체를 제거 | 폭풍 예보날 캠핑 장소 변경 | 리스크가 아예 消滅. 계획 변경 필요 |
| **Transfer (전가)** | 임팩트를 제3자에게 넘김 | 자동차 보험 가입 | 리스크는 존재, 임팩트만 이전 |
| **Mitigate (완화)** | 확률 또는 임팩트를 줄임 | 비 올 수 있는 등산에 우비 챙김 | 리스크는 남아있음, 크기만 줄어듦 |
| **Accept (수용)** | 그냥 받아들임 | 비 맞으면 맞는 거지 | Active(예비비 설정) / Passive(그냥 수용) |
| **Escalate (상향)** | 상위 권한자에게 넘김 | PM 권한 밖의 리스크일 때 | 프로젝트 범위 초과 리스크 |

#### Opportunity (긍정적 리스크) 대응

| 전략 | 한 줄 설명 | 비유 | 핵심 |
|---|---|---|---|
| **Exploit (활용/극대화)** | 기회를 반드시 실현시킴 | 좋은 딜 소식에 즉시 선주문 | 기회를 100% 확정. **Threat에 절대 쓰지 않음** |
| **Enhance (강화)** | 기회의 확률/임팩트를 높임 | 당첨 확률 높이려고 복권 더 삼 | 기회가 생길 가능성을 키움 |
| **Share (공유)** | 기회를 더 잘 잡을 수 있는 제3자와 공유 | 조인트벤처 구성 | Transfer의 Opportunity 버전 |
| **Accept (수용)** | 기회 오면 잡고 아니면 말고 | - | 적극적으로 추구하지 않음 |
| **Escalate (상향)** | 상위 권한자에게 넘김 | - | 프로젝트 범위 초과 기회 |

> ⚠️ **Exploit = 기회 전용**. Threat 문제에 나오면 무조건 오답
> Transfer(위협) ↔ Share(기회) / Mitigate(위협) ↔ Enhance(기회) 대칭 관계

### 헷갈리는 Plan 용어 구분

| 용어 | 정의 |
|---|---|
| **Mitigation Plan** | Mitigate 전략의 구체적 실행 내용 (Risk Register 안에 기록) |
| **Contingency Plan** | 리스크가 **실제 발생했을 때** 실행할 비상 계획 |
| **Fallback Plan** | Contingency Plan도 실패했을 때 최후 대응 |
| **Risk Response Plan** | 모든 리스크 대응 전략 모음 (Risk Register 안에 포함) |

```
포함 관계:
Mitigation Plan ⊂ Risk Response Plan ⊂ Risk Register
```

> Mitigation Plan = 별도 문서 아님. Risk Register 안에 기록되는 내용.

# 변경 관리 (Change Management)

### 변경 요청 프로세스

> 핵심: **"approved"** 라는 단어가 보이면 → 승인 단계는 이미 끝난 것. 다음 액션은 **계획 문서 업데이트**.

```
변경 필요 발생
    ↓
Change Request 작성 (문서화)
    ↓
CCB / Steering Committee 검토 및 승인/거부
    ↓
✅ 승인되면 → 계획 문서 업데이트
    ↓
팀에 공지 및 실행
    ↓
필요시 Activity duration 재계산, 일정/비용 재산정
    ↓
변경 결과 모니터링
```

| 상황 | 정답 액션 |
|---|---|
| 변경 필요 발생 | Change Request 작성 |
| Change Request 제출 후 | CCB/Steering Committee 승인 대기 |
| **이미 Approved** | **계획 문서 업데이트 → 실행** |
| PM 혼자 변경 결정 | ❌ 항상 오답 |

# 리더십 & 팀 관리 (Leadership & Team)

### 두 프레임워크 구분 (⚠️ 혼동 주의)

> PMP 시험에서 리더십 관련 문제는 두 개의 완전히 다른 프레임워크에서 출제됨. 섞이면 틀림.

| | **Behavioral Orientation** | **Situational Leadership Style** |
|---|---|---|
| **출처** | McClelland 동기이론 기반 | Hersey & Blanchard |
| **질문** | "이 PM은 뭐에 의해 움직이나?" | "이 PM은 상황에 따라 어떻게 리드하나?" |
| **본질** | PM의 **성격/동기 성향** (고정적) | PM이 선택하는 **리드 방식** (상황 적응) |
| **시험 키워드** | **"behavioral orientation"** | **"leadership style"**, "team maturity" |

#### 덩어리 1: Behavioral Orientation (동기 성향)

| 성향 | 동기 | 시험 선지 형태 |
|---|---|---|
| **Directive** | 명확한 결과·완료, 간결한 소통 | "implement communication channels" |
| **Affiliation** | 관계·소속감 | "build rapport and relationships" |
| **Achievement** | 성취·탁월함 | "motivate with achievement awards" |
| **Power** | 영향력·통제 | "influence decisions and control delivery" |

> ❗ **Directive ≠ Control** — PMI에서 통제 = Power orientation. Directive = 명확한 커뮤니케이션 + 태스크 완료 집중
> 참고: PMI Learning Library - Successful Motivational Techniques for Virtual Teams

#### 덩어리 2: Situational Leadership Style (상황적 리더십)

| 스타일 | 팀 상태 | 시험 선지 형태 |
|---|---|---|
| **Directing (Telling)** | 낮은 역량 + 높은 의욕 | "give clear instructions, closely supervise" |
| **Coaching** | 낮은 역량 + 낮은 의욕 | "explain why, develop skills" |
| **Supporting** | 높은 역량 + 낮은 의욕 | "involve in decisions, provide support" |
| **Delegating** | 높은 역량 + 높은 의욕 | "empower, self-organizing team" |

---

### Servant Leadership — Hybrid/Agile 전환 시

> 핵심: **Predictive → Hybrid/Agile 전환 시 PM 역할이 바뀜** = Servant Leadership

```
Predictive PM   = 내가 계획·통제·지시
                        ↓ 전환
Hybrid/Agile PM = 팀이 스스로 움직이게 장애물 제거·지원
                = Servant Leadership
```

- **킬러 키워드**: enable, empower, remove impediments, support, focus on mission
- Agile/Hybrid 환경 + "팀이 미션에 집중하게" → 항상 Servant Leadership
- A(Scrum), C(Lean) 같은 **방법론 선지**는 리더십 문제에서 항상 오답

### 동기부여 이론 식별

> 상세 내용 → `2-2. 팀 - Team.md` §5 동기부여 이론

| 이론 | 핵심 | 킬러 키워드 |
|---|---|---|
| **Expectancy** (Vroom) | "이걸 하면 저걸 받는다" 성과-보상 연결 | if-then 보상, 금전/휴가 제시 |
| **Achievement** (McClelland) | 성취/친화/권력 욕구로 동기부여 | 성취감, 도전적 목표, 인정 |
| **Maslow** | 욕구 5단계, 하위 충족 후 상위 욕구 | 안전/소속/존중/자아실현 |
| **Herzberg** | 동기요인 vs 위생요인 | 급여 올려도 동기 안 오름 |
| **Contingency** (Fiedler) | 리더십 효과 = 상황에 따라 다름 | 상황에 맞는 스타일 |
| **McGregor X/Y** | X=통제, Y=자율·신뢰 | PMP는 무조건 Theory Y |

> ⚠️ **팀 동기부여에서 Competitive environment = ❌ 항상 오답**
> 이유: 일부에게만 효과적, 팀워크 파괴, 불안감 조성

# 이해관계자 관리 (Stakeholder Management)

### 저항 예방 — Early Engagement (선지 해석 여지 주의)

> 핵심: **저항 예방 = 처음부터 함께 결정하기** (동의 받기가 아님)

#### 문제 패턴
> 이해관계자들이 새 기술 채택에 저항 → PM이 미리 했어야 할 것?
> → 정답: **D. Engage stakeholders as early as possible**

#### 헷갈리는 선지 해석

| 선지 | 구조 | 왜 틀렸나 |
|---|---|---|
| **A. Invest in communication** | PM → 이해관계자 (일방적) | 소통 ≠ 참여. 결정에서 배제됐으면 저항함 |
| **C. Propose an agreement** | PM 제안 → 이해관계자 사인 | 일찍 해도 top-down. 진짜 buy-in 아님 |
| **D. Engage early** ✅ | PM ↔ 이해관계자 (함께 결정) | 처음부터 참여 → 주인의식 → 저항 없음 |

```
Communication = 정보 전달 (일방향 가능)
Engagement    = 참여·의견 반영 (쌍방향 필수) ← PMP가 원하는 것

Agreement = Engagement의 결과물이지 대체재가 아님
```

# 조달 관리 (Procurement Management)

### 조달 문서 4종 비교 (RFI / RFQ / RFP / RFC)

> 핵심: **신규 벤더에게 줄 가장 포괄적인 문서 = RFP**

| 문서 | 정식 명칭 | 언제 씀 | 정보량 | 특징 |
|---|---|---|---|---|
| **RFI** | Request for Information | 벤더 탐색 초기 "어떤 업체 있나?" | ⭐ | 비공식. 특정 측면만 다룸. 탐색용 |
| **RFQ** | Request for Quote | 뭘 살지 확정됨. 가격만 물어볼 때 | ⭐⭐ | 단가/견적 위주. 범위 정보 적음 |
| **RFP** | Request for Proposal | 문제는 있는데 솔루션 불확실 | ⭐⭐⭐⭐ | 가장 포괄적. 요구사항+인수기준 전부. 가장 공식적 |
| **RFC** | Request for Contract | 계약 체결 단계 | ⭐⭐ | RFP를 참조함. 범위 정보 적음 |

```
정보량: RFP > RFC ≈ RFQ > RFI
공식성: RFP > RFC > RFQ > RFI
```

#### 문제 패턴 1: 납품물 인수 기준 미달 → 벤더 검증

> "팀이 외주 납품물 받았는데 인수 기준 미달 → PM이 벤더와 검증하려면 뭘 참고?"
> → 정답: **C. Statement of Work (SOW)**

- SOW = 계약 시 합의한 문서. **인수 기준이 여기 명시**됨.
- RFP = 계약 전 문서. 계약 후엔 SOW가 기준.
- RFQ = 가격 견적용. 인수 기준 없음.

#### 문제 패턴 2: 교체 벤더에게 줄 문서

> "실행 단계에서 벤더 교체 → 새 벤더한테 프로젝트 정보 줘야 함 → 어떤 문서?"
> → 정답: **C. RFP**

- 새 벤더는 프로젝트 컨텍스트를 처음 보는 것 → 가장 많은 정보가 필요
- RFP = 요구사항 + 인수 기준 + 범위 전부 포함
- RFQ/RFC = 범위 정보 부실. 새 벤더한테 못 줌.

### SOW (Statement of Work)

| 특징 | 내용 |
|---|---|
| **역할** | 계약 시 벤더와 합의한 작업 범위·납품물·인수 기준 명세 |
| **시점** | 계약 체결 후 (= 합의 완료 상태) |
| **포함 내용** | 작업 범위, 납품 일정, 인수 기준, 기술 요구사항 |
| **시험 활용** | 납품물 품질 분쟁 시 → SOW가 검증 근거 |

### 다국가 프로젝트 — 문화 차이 사전 평가

> 핵심: **PM은 관리자한테 먼저 달려가기 전에 스스로 상황을 파악하고 주도해야 함**

#### 문제 패턴
> 5개국 프로젝트에 복장/사무실 지침 발행 → 3개국 저항/오해/법적 문제 → 미리 뭘 했어야?
> → 정답: **B. Evaluated cultural and personal traits before issuing directive**

#### A vs B — 단골 함정

| | A. 시니어 매니저에게 예외 요청 | B. 문화 사전 평가 ✅ |
|---|---|---|
| **시점** | 사후 (문제 생기면) | 사전 (지침 발행 전) |
| **주체** | 관리자 의존 | PM이 직접 주도 |
| **접근** | Reactive | Proactive |

> 개인 선호도(D) = 개인 레벨 / 문화 차이 = 국가·집단 레벨 — 스케일이 다름
